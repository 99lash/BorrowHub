## 🔍 Detailed Findings

### 🟠 High Severity

#### 1. Android Code Obfuscation Disabled
*   **Location:** `mobile-app/app/build.gradle.kts`
*   **Vulnerability:** The release build configuration has `isMinifyEnabled = false`.
*   **Impact:** The compiled APK will contain plain Java/Kotlin code that is easily readable via decompilers. This allows attackers to reverse-engineer the application logic, find API endpoints, and understand the internal workings of the app.
*   **Recommendation:** Set `isMinifyEnabled = true` in the `release` block and configure ProGuard/R8 rules.

### 🟡 Medium Severity

#### 2. Insecure Token Storage (Android)
*   **Location:** `mobile-app/app/src/main/java/com/example/borrowhub/data/local/SessionManager.java` (or similar)
*   **Vulnerability:** Authentication tokens are stored in plain text using standard `SharedPreferences`.
*   **Impact:** On rooted devices or via certain exploit vectors, the auth token can be extracted from the app's private directory, allowing session hijacking.
*   **Recommendation:** Migrate to `EncryptedSharedPreferences` for storing sensitive credentials like API tokens.

#### 3. Non-Expiring API Tokens (Laravel)
*   **Location:** `backend-api/config/sanctum.php`
*   **Vulnerability:** The `expiration` setting for Laravel Sanctum is set to `null`.
*   **Impact:** API tokens issued to mobile devices never expire. If a token is stolen, it remains valid indefinitely unless manually revoked by an administrator or the user logging out.
*   **Recommendation:** Set a reasonable expiration time (e.g., `10080` for 7 days or `43200` for 30 days) in `config/sanctum.php`.

#### 4. Permissive Network Security Configuration (Android)
*   **Location:** `mobile-app/app/src/main/res/xml/network_security_config.xml`
*   **Vulnerability:** `cleartextTrafficPermitted="true"` is enabled for several IP ranges (`127.0.0.1`, `10.0.2.2`, `192.168.1.5`).
*   **Impact:** While useful for development, this allows unencrypted HTTP traffic. If the app is deployed with this configuration, it is vulnerable to Man-in-the-Middle (MITM) attacks.
*   **Recommendation:** Use `debug-overrides` in the configuration to allow cleartext only during development, or ensure the production version strictly enforces HTTPS.

#### 5. Application Backup Enabled (Android)
*   **Location:** `mobile-app/app/src/main/AndroidManifest.xml`
*   **Vulnerability:** `android:allowBackup="true"` is set.
*   **Impact:** This allows users (or attackers with physical access) to back up application data, including the private `SharedPreferences` (which contains the auth token), via `adb backup`.
*   **Recommendation:** Set `android:allowBackup="false"`.

### 🔵 Low Severity

#### 6. Weak Default Passwords
*   **Location:** `backend-api/config/auth.php` and `.env.example`
*   **Vulnerability:** `AUTH_DEFAULT_USER_PASSWORD` is set to `borrowhub123` (or similar).
*   **Impact:** This weak password is used for password resets and default users, making accounts susceptible to brute-force attacks if not changed immediately.
*   **Recommendation:** Implement a more robust password reset mechanism that generates unique random strings or forces a password change upon the first login.

#### 7. Insecure Password Hints
*   **Location:** `mobile-app/app/src/main/res/values/strings.xml`
*   **Vulnerability:** Contains a hint like `password_hint">Enter your password (123)`.
*   **Impact:** Encourages users to use extremely weak passwords.
*   **Recommendation:** Remove hardcoded password hints from the UI strings.

## ✅ Positive Security Practices Identified

*   **Concurrency Control:** `TransactionService.php` correctly uses `DB::transaction()` and `lockForUpdate()` to prevent race conditions and over-borrowing of items.
*   **Validation:** Strict use of **Laravel Form Requests** ensures that input is validated before reaching the controller logic.
*   **Authentication:** Uses **Laravel Sanctum** for token-based authentication, which is an industry standard for SPAs and mobile apps.
*   **Architecture:** Adherence to the **Service-Repository pattern** prevents "Fat Controllers" and keeps business logic isolated and testable.
*   **Data Protection:** Sensitive fields like `password` are correctly listed in the `$hidden` array of the `User` model.
*   **Audit Logging:** A dedicated `LogService` tracks sensitive actions like user creation, student imports, and item deletions.

## 🛠 Remediation Plan (Recommended Next Steps)

1.  **Immediate:** Enable `isMinifyEnabled` in the Android release build (`build.gradle.kts`).
2.  **Short-term:** Update `SessionManager.java` to use `EncryptedSharedPreferences`.
3.  **Short-term:** Set a TTL for API tokens in `backend-api/config/sanctum.php`.
4.  **Governance:** Update the password reset logic to eliminate weak default passwords.
